# phase-implementation

## 目的

system_design.md と CLAUDE.md のフェーズ計画に従い、1フェーズ分を「設計確認→実装→Docker起動→動作確認→定型レポート」の手順で、動作確認可能な状態まで完了させる。

## 担当範囲

- 対象フェーズの実装範囲確定(project-managerの担当と重複する場合はそちらを優先)
- Backend実装(model / schema / service / router)
- Frontend実装(features配下、一覧画面はTanStack Table)
- DBマイグレーション作成(Alembic)・初期データ投入
- Docker環境での起動・動作確認
- 定型フォーマットでの完了報告

## 手順

1. **設計確認**
   - system_design.md と CLAUDE.md を読み、今回実装するフェーズの範囲(対象テーブル・API・画面)を特定する。
   - 既存ファイル構成を確認し、既存コードを壊さないことを確認する。
   - 対象フェーズに無い機能は実装しない(先読み実装をしない)。

2. **Backend実装**
   - `app/<feature>/model.py`(SQLAlchemy) / `schema.py`(Pydantic) / `service.py`(業務ロジック) / `router.py`(エンドポイント)の順で作成する。
   - routerに業務ロジックを書かず、service層に分離する。
   - 認証が必要なエンドポイントには `get_current_user` を適用する。
   - DBテーブルを追加・変更する場合は Alembic マイグレーションを1ファイル作成する(`upgrade`/`downgrade`両方を実装)。初期データ投入が必要な場合はマイグレーション内で `bulk_insert` する。

3. **Frontend実装**
   - `features/<feature>/` 配下に型定義・APIクライアント・画面コンポーネントを配置する。
   - 一覧画面は必ず TanStack Table を使用し、ソート・フィルタ・ページングを標準実装する。
   - API通信は `lib/api` のクライアントに集約し、各featureのAPIモジュールはそれを呼ぶだけにする。
   - any型を使用しない。

4. **Docker起動・動作確認**
   - `docker compose up -d --build` でビルド・起動する。
   - ビルド/起動エラーが出た場合は原因(ポート競合、ネットワーク、DB初期化失敗など)を特定して修正する。DBボリュームが古い認証情報で初期化されている場合は、当該フェーズで新規に作成したボリュームに限り `docker compose down -v` で再作成してよい(本物の運用データが入っているボリュームには絶対に行わない)。
   - `curl http://localhost:8000/health` で起動確認する。
   - 追加したAPIを実際に `curl` で呼び出し、レスポンス(正常系・認証エラー等)を確認する。
   - `docker compose exec db mysql ...` でテーブル作成・初期データ投入結果を確認する。
   - 文字列が `?????` のように見える場合はマルチバイト文字のターミナル表示の問題であることが多いため、`--default-character-set=utf8mb4` を付けて再確認する(データ破損と早合点しない)。

5. **完了報告**
   - 以下の定型フォーマットで出力する。

```text
# 実装内容
# 作成ファイル
# 変更ファイル
# 動作確認方法
# 次フェーズ
```

   - 最後に「▶ 次へ進める場合は「続き」と出力する」を添える。

## 入力

- system_design.md
- CLAUDE.md
- 現在の実装状況(既存ファイル)
- 対象フェーズ番号

## 出力

- 実装コード(Backend / Frontend / マイグレーション)
- 動作確認済みのDocker環境
- 定型フォーマットの完了レポート

## 注意点

- 一度に複数フェーズを実装しない
- 設計書にない機能を追加しない
- 未完成コードで終了しない
- 動作確認(curl等の実コマンド実行結果)なしで完了報告をしない
- ポート競合や認証エラーなどの環境起因の問題は、原因を特定したうえで最小限の修正(ポート番号変更、不要な古いボリューム削除など)に留める
